# ppt神器

## 总闸门

- 先定内容，再生成图片；不做 PPTX。
- 每页成品必须是 image 模型生成的单页 16:9 PNG。
- 默认用户可见交付只有 `页面图片/*.png` 和 `演讲提纲.md`。
- 引用与假设说明默认合并在 `演讲提纲.md` 末尾；只有用户明确要求，或正式学术答辩/报告引用很多时，才额外生成 `引用与假设说明.md`。
- 两次确认不能合并：先确认页序与内容逻辑，再生成两张风格样张；用户确认样张后才批量生成剩余 PNG。
- 第一次确认前必须先判断材料状态和场景契约。页序草案是第一个用户可见确认物，不是第一步逻辑。

## 按需读取

不要一开始读取所有文件。先判断用户当前意图，再只读对应资源：

| 当前任务 | 读取什么 |
| --- | --- |
| 用户问怎么用、为什么这样设计、当前状态 | 只读本文件；必要时读 `references/workflow.md` 的相关小节 |
| 用户要开始或继续制作 PPT | `references/workflow.md` |
| 页序已确认，要生成两张样张或诊断图片质量 | `references/page-standards.md` + `assets/图像提示词模板.md` + 当前成品包的 `场景视觉简报.md` 和当前页 `页面规格.json`，并读取套系一致性锚点 |
| 需要写内部文件 | 只读当前要填写的模板，不要一次性读取全部 assets |
| 用户要测试 prompt | `references/examples.md` |
| 用户明确要单独引用说明 | `assets/引用与假设说明模板.md` |

不要因为生成图片就复制整份标准文件；图像提示词只抽取当前页必要字段。

## 执行状态

执行工作流时按状态推进：

1. `待确认页序`：先完成材料分流和场景契约，再只给页序与内容锁定草案，不生成图片。
2. `待确认首图`：只生成第 1 页和 1 张代表性内容页样张，展示后停止。
3. `批量生成中`：用户明确确认样张后，再生成剩余页面。
4. `最终交付`：只主展示 `页面图片/` 和 `演讲提纲.md`。

即使用户一开始说“全部生成”，也默认走两次确认，除非用户明确要求跳过确认。

## 信息分层

上游决定，下游引用，不重复解释：

- 材料分流：判断用户属于六种材料状态中的哪一种：只有主题、有大纲、有完整文档、有散乱素材、有旧 PPT、有数据表。
- 场景契约：确定使用场合、听众、页数、时长、证据密度、内容边界和交付物。
- 页序与内容锁定草案：第一次用户确认物，用来确认页面顺序、页面角色、听众问题、结论、核心内容关系和来源状态。
- PPT 内容蓝图：在页序确认后展开“讲什么、为什么这样讲、每页有哪些内容块”。
- 内容关系简报：说明每页内容如何成立、观众为什么能听懂；它嵌在内容蓝图和页面规格里，不是单独交付文件。
- 场景视觉简报：决定整套 PPT 应该像哪类真实材料、什么气质和信息密度。
- 逐页视觉导演决策：决定每页用什么视觉策略表达内容关系；它写入页面规格。
- 页面规格：把内容、来源、内容关系和视觉决策整理成每页 image 生成字段。
- 图像提示词：压缩转交给 image 模型，不承担重新设计流程。

## 来源底线

所有内容必须标注来源状态：来自用户材料、来自已核验引用、需要从表格计算、模拟假设、待核验。

学术场景在风格确认前必须先写引用框架和待核验来源，但不做广泛文献检索。不能编造 DOI、样本量、年份、题名、精确数字或不存在的论文。

## 视觉底线

- 页面要像真实用户明天就能汇报的材料，不是海报、封面、模板样机或素材拼贴。
- 视觉方向优先写“应该成为什么真实材料”，不要用大量负向规则驱动 image。
- 首图或风格样张必须是两个独立单页 PNG，不是缩略图、风格板、mockup 或多页拼排。

## 完成检查

交付前确认：

- `页面图片/` 页数与用户要求一致。
- 每页 PNG 来自 image 模型。
- `演讲提纲.md` 已包含讲稿准备、引用与假设说明。
- 样张确认前没有提前生成剩余页面。
- 最终回复只主展示 `页面图片/` 和 `演讲提纲.md`。
